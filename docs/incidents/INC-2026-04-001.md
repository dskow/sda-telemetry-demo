# INC-2026-04-001 -- Telemetry consumer wedged after upstream schema drift

- **Severity**: 2
- **Detected**: 2026-04-14 16:32 UTC
- **Resolved**: 2026-04-14 17:48 UTC
- **Duration**: 1h 16m
- **Reporter**: on-call (synthetic example for portfolio demo)

## Summary

The `sda-telemetry` consumer stopped advancing its committed offset on partition 1
of `rso.telemetry`. The same record was reprocessed in a tight loop, producing
roughly 4,000 ERROR log lines per minute. The catalog continued serving REST reads
but was no longer applying new telemetry, so the data steadily aged. Severity 2.

## Impact

- Catalog age on partition-1 RSOs grew from <5s to ~76m before resolution.
- Operator dashboards showed stale lat/lon values for ~30% of tracked objects.
- No data loss; once the underlying defect was patched, replay caught up in 11m.

## Root cause

A producer upstream of NiFi began emitting `altitudeKm` as a quoted string
(`"408.0"`) instead of a JSON number, after a partner system was upgraded.

The consumer used a plain `JsonDeserializer<TelemetryUpdate>`. When deserialization
failed, Spring Kafka's default container behavior was to retry the same record
indefinitely (poison pill), because the deserialization exception happened
*before* the listener method was invoked, so the configured listener-level error
handler never saw the record.

The container had no `ErrorHandlingDeserializer` wrapper and no
`DeadLetterPublishingRecoverer`, so there was no escape valve for a malformed
record.

## Detection

Consumer-lag alert (A2 in the runbook) fired 11 minutes after onset. The on-call
engineer found the repeating ERROR loop in logs and identified the bad record by
its offset.

## Remediation

1. Short-term (16:48 UTC): manually advanced the consumer-group offset past the
   single bad record using `kafka-consumer-groups --reset-offsets --to-offset
   <next>`. Consumer resumed.
2. Long-term (commit landed 16:55 UTC, deployed 17:48 UTC):
   - Wrapped `value.deserializer` with `ErrorHandlingDeserializer`, with the
     real `JsonDeserializer` as the delegate. Deserialization failures now
     surface as a `DeserializationException` inside the listener, where the
     container's error handler can intercept them.
   - Configured `DeadLetterPublishingRecoverer` to publish bad records to
     `rso.telemetry.dlq` with the original headers preserved.
   - Configured `DefaultErrorHandler` with `FixedBackOff(1s, 3)` so transient
     failures retry briefly before being routed to the DLQ.
   - Marked `DeserializationException` and `IllegalArgumentException` as
     non-retryable -- they will never succeed on retry.

## Why the fix is correct

Spring Kafka's listener-level error handlers operate on records that successfully
deserialized. To handle deserialization failures, the deserializer itself must
not throw -- it must return a special record that the listener layer can
recognize. `ErrorHandlingDeserializer` does exactly this, allowing the same
`DeadLetterPublishingRecoverer` to handle both deserialization failures and
listener-method exceptions uniformly.

## Lessons learned

1. **Treat partner schemas as untrusted by default.** The producer-side change
   was announced via an email thread that the consumer team did not see. We are
   not adding contract-test gates against partner schemas (out of scope for this
   project), but the DLQ now contains the bad record for later analysis instead
   of jamming the consumer.
2. **Listener-level error handlers do not catch deserialization errors.** This
   is documented in the Spring Kafka reference but is easy to miss. The runbook
   now lists schema drift as the first triage step for `dlq-rate-high`.
3. **The catalog is idempotent on `(rsoId, sensorId, observedAt)`,** so the
   offset reset was safe. We confirmed this property in the unit tests
   (`RsoCatalogTest.applyTelemetryRejectsStaleObservation`,
   `differentSensorsTrackedIndependently`).
